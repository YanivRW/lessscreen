# Supabase & data model — detailed notes

Loaded when working with the schema, repositories, or models. High-level config
is in the root CLAUDE.md.

## Tables (public schema)

- **`profiles`** — one row per user. Has `invite_code` (6-char, auto-generated by
  a trigger on signup). A `display_name` field exists but has no UI yet.
- **`friends`** — two rows per friendship (A→B and B→A). Mutual, auto-accept via
  invite code, so both users see each other immediately on add.
- **`daily_usage`** — one row per user per day, stores `total_minutes` for that day.

## RLS

All three tables have Row Level Security enabled. Friends can read each other's
`daily_usage` rows. Profiles are readable by all authenticated users (needed for
invite-code lookup before a friendship exists).

## Auth trigger

`on_auth_user_created` fires after every signup and auto-creates a `profiles` row
with a random 6-char `invite_code`. Uses `md5(id || now())` — deliberately avoids
the `pgcrypto` extension so it works on a default Supabase project.

## Email confirmation

Currently OFF in Supabase Auth settings — users sign up and are immediately
signed in. Re-enabling it means users must click an email link before first
sign-in.

## Schema-change discipline

`supabase/schema.sql` is the source of truth; it gets run in the Supabase SQL
Editor. When adding data-model changes for the product direction (blocking
schedules, the future friend-lock feature), design the tables/columns **once** so
we don't migrate repeatedly. Anticipated additions to think through together
before writing SQL:

- **Blocking schedules** — likely per-user, per-app, with time windows. Decide
  local-only vs. Supabase-synced early (syncing enables the friend-lock feature
  later but adds complexity).
- **Friend-lock (future)** — a friend sets a secret passcode that remotely
  restricts one of your apps. Whatever the blocking schedule model is, leave room
  for a lock that's *owned/controlled by a different user* than the one being
  blocked, and for storing a hashed passcode the locked user can't read.
- **Per-app usage for friends** — current `daily_usage` only stores a daily
  total. Per-app friend breakdown (a planned feature) would need a more granular
  row shape; weigh that against payload size and RLS complexity.
